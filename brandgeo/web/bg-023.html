How We Audit a Site for AI-Readiness Without a Paid Crawling API

No headless browser, no paid crawling service. A sitemap, robots.txt respect, and regex-based extraction of the structural signals that actually matter for GEO. Here is exactly how it works, and the one real limit we built it to disclose.

Free by design, not free by accident

When we built the crawler behind BrandGEO's AI SEO page audits, the first decision was what it would depend on. A lot of crawling infrastructure in this space runs on a paid third-party crawling API or a headless browser cluster. Ours runs on a client's own sitemap.xml and a plain server-side fetch, on purpose, so a page-level AI-readiness audit doesn't carry a per-page cost that has to get passed on somewhere.

Discovery: sitemap first, homepage links as the fallback

The crawler tries three sitemap locations in order, /sitemap.xml, /sitemap_index.xml, /sitemap-index.xml, and walks sitemap indexes recursively up to a depth of 2 so a large site's nested sitemap structure resolves correctly, with a visited-set guarding against a circular reference looping forever. If no sitemap exists at all, the crawler falls back to extracting same-host links straight from the homepage, so a small site without a sitemap still gets a usable page list rather than an empty result.

Every discovered URL is filtered three ways before it's crawled: it has to sit on the same host as the domain being audited, it has to be allowed by a minimal robots.txt parser that respects Disallow rules for BrandGEO's own user agent and for *, and it's deduplicated after stripping any URL fragment. The whole list is capped at 25 pages by default, enough for a real audit sample without turning a single request into an unbounded crawl.

Extraction: regex, not a DOM, on purpose

Each page fetch is bounded to 8 seconds and 1.5MB, a page larger than that is truncated to the first 1.5MB rather than dropped outright, so an unusually large page still contributes partial signal instead of nothing. Title and meta description are pulled with an og:title-first, <title>-fallback approach, matching what an AI engine or search crawler is more likely to actually read.

The technical signals that matter most for GEO are detected directly: whether the page carries any JSON-LD structured data, whether that JSON-LD includes FAQ schema specifically, whether the page has tables, whether it has lists, and how many H1 headings it uses. Body content is converted to a lightweight markdown-like text, headings become ##/### markers, list items become - bullets, so the page-scoring model downstream can judge structure from readable text rather than raw HTML tag soup.

The limit we built in on purpose, and disclose rather than hide

This approach has a real ceiling: a plain server-side fetch reads the HTML a server sends, it does not execute JavaScript. A site that renders its actual content client-side, after the initial HTML loads, will crawl thin or empty with this method. We didn't design around that limit quietly, the crawler's entry point is deliberately structured as a swappable provider, crawlSite() calls a sitemap-and-fetch implementation today with a commented-out slot for a JS-rendering provider (Firecrawl) to drop in later for exactly this case, without any caller needing to change.

What this means in practice today

If your site is server-rendered or static, standard for most marketing sites and blogs, this crawler reads it accurately. If your site is a client-side-rendered single-page app with no server-rendered content, an audit run today may under-report your actual page content until the JS-rendering provider ships. We'd rather tell you that limit exists than have you discover it as an inexplicably low score.

What this means when you're comparing AI SEO or GEO audit tools

A free, sitemap-based crawl and a paid, headless-browser crawl are solving different problems, and neither is universally better, the honest question is whether a tool's crawling method matches your site's actual architecture. Ask what a vendor's crawler actually reads, whether it executes JavaScript, what it does when a page is oversized or slow, and whether they've told you where the method's real limits are before you found out the hard way.
