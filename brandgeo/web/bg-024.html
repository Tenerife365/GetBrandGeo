Guardrails Without a CAPTCHA

Our free public AI Visibility audit has no CAPTCHA and no login wall, on purpose, because friction loses real visitors. Here is what actually protects it: a honeypot that lies to bots, and IP rate limiting that never stores a real IP.

A CAPTCHA is a tax on every real visitor to stop a minority of bots

BrandGEO's public Instant Audit endpoint is anonymous by design, anyone can type a domain and get a free AI Visibility read without creating an account. That openness is also exactly what a scraper or abuse script would want to hit repeatedly. The obvious answer is a CAPTCHA, and we deliberately didn't add one, because a CAPTCHA punishes every legitimate visitor to slow down a much smaller group of automated ones. guardPublicRequest in _prospect_guard.js is what we built instead.

A hidden field that lies to bots instead of rejecting them

The first layer is a honeypot: a form field real users and real browsers never fill in, because it's hidden from view, but a naive automated form-filler often fills every field it finds. When that field arrives non-empty, the request doesn't get an error response. It gets a 202 with a shape that looks like a normal success, { token: null, status: 'pending' }, logged quietly on our side and silently dropped. A bot that receives an honest rejection can adapt; a bot that receives a fake success has no signal to learn from.

Rate limiting an IP without ever storing one

The second layer is a hard cap: 3 public audits per IP per day. The part worth disclosing is how the IP is handled. hashIp runs SHA-256(ip + server-side pepper) and only that hash is ever written to the database, never the plaintext address. The pepper is a server-side secret, so the hash can't be reversed or matched against a leaked list even if the hashes themselves were exposed. The code comment attached to this function states the reasoning plainly: it exists "to rate-limit, never stored/exposed as a real IP," citing GDPR data minimization directly as the reason, not as an afterthought bolted on for compliance theater.

Why hash instead of just not logging IPs at all

Rate limiting by IP requires comparing today's request to previous ones from the same address, which means some form of the IP has to be stored somewhere. A salted hash gives us that comparison capability, "is this the same visitor as 4 hours ago" without persisting anything that identifies a real person or address. It's the minimum data actually needed for the guardrail to function, not the maximum data convenient to collect.

The layers that run before either of those

An explicit origin allowlist, 5 origins covering the marketing site, the dashboard, and local development, rejects any request whose Origin header doesn't match, before the honeypot or rate limit are even checked. A lightweight domain-format validator rejects obviously malformed input with a regex, no real DNS lookup required, cheap enough to run on every request without adding latency. And a shared internal secret lets BrandGEO's own backend services call the same endpoint without hitting the public IP cap meant for anonymous visitors, a legitimate bypass for a legitimate caller, not a loophole.

What this means for any "free, no signup" tool you evaluate

A public tool with zero friction and zero protection is a liability; a public tool with heavy friction defeats its own purpose. The layers worth checking for in any vendor's free-tier tooling are the ones that cost real visitors nothing, an allowlist, a honeypot, format validation, while reserving the layer that does cost something, a rate limit, for the case that's actually abuse. And if the tool collects your IP to do it, it's worth asking whether it's stored in a form that could ever be reversed back to you.
