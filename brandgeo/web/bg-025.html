Why One Spend Limit Wasn't Enough

A monthly euro budget sounds like it should be enough to protect a free tool's cost. It isn't, on its own, because it can't see a fast burst coming until the burst has already spent the budget. Here's why we run two independent circuit breakers instead of one.

A budget check alone has a blind spot in its first few minutes

BrandGEO's free public Instant Audit tool spends real API cost on every request. A monthly euro budget, checked before each call, is the obvious control, and we have one, €200 by default, configurable per deployment. But a monthly budget alone has a real gap: a sudden burst, a bug in a script hammering the endpoint, a coordinated abuse attempt, wouldn't trip that check until enough cost had already accumulated within the month. In the first several minutes of a fast burst, the monthly total simply hasn't caught up yet.

Two breakers, scoped to two different failure shapes

checkMonthlyBudget sums estimated_cost_eur across every prospecting audit created since the start of the current calendar month, and compares that live sum against the configured ceiling. It deliberately does not maintain a separate running counter, it derives the total on the fly from the same table every time, the same pattern the product's own usage dashboard uses to report cost, so there is exactly one source of truth for "how much have we spent," not two numbers that could quietly drift apart.

checkGlobalHourlyLimit is a different shape of check entirely: a global cap of 100 requests per rolling hour, across all anonymous callers combined, since a public prospecting endpoint has no per-client identity to scope a limit to. It exists specifically to catch what the monthly budget structurally cannot: a fast spike in the first few minutes, before enough individual calls have accumulated cost for the monthly total to notice.

Why the hourly limit is set generously

100 requests an hour is a high bar for organic traffic to a free audit tool. That's deliberate: this breaker isn't meant to be the primary cost control, the monthly budget is. Its entire job is catching runaway abuse fast enough that the monthly budget never has to absorb the full cost of a multi-minute burst before anything reacts. Setting it too tight would start rejecting real traffic; setting it generously and pairing it with the budget check catches the failure mode neither one catches alone.

Not a one-off pattern

This two-layer shape isn't unique to the public prospecting tool. The authenticated side of the product uses the same principle at a different scope, a per-client hourly row limit protecting individual client accounts from a runaway integration or script, distinct from the account-level plan limits that cap normal usage. Different endpoints, same underlying idea: a slow, precise control for the common case, and a fast, generous breaker for the failure mode the precise control can't see coming in time.

What this means when you're building or evaluating cost controls

A single spend limit checked before every call feels sufficient until the exact week a burst runs faster than that limit can react. Worth asking of any system carrying real per-request cost: is there one control or two, and if the answer is one, what happens in the specific minutes between a burst starting and that one control finally tripping? That gap is exactly what a second, differently-scoped breaker is for.
