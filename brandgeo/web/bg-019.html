Why Our Scorer Returns Null Instead of a Rank It Can't Prove

BrandGEO only reports a rank when an AI engine's answer makes one of three explicit claims. Everything else, including a sentence position that looks like a plausible rank, returns null. Here is the code and the tests behind that decision.

A "few good options" isn't a ranking, even if your brand is bullet one

A bulleted list of "a few good caterers in Bucharest" and a genuinely ranked "top 3 caterers, in order" look identical to a parser that only checks for bullet characters. Score the first as if it were the second, and you have manufactured a rank the AI engine never actually claimed. That is a subtler failure than the extraction bugs we wrote about last week, because it does not add a wrong name to a list, it adds a wrong number to a real one.

The rule we settled on, after finding this exact failure in our own AI Visibility Score pipeline, is narrower than it might sound: a brand only gets a numeric position when the engine's answer makes one of three specific, checkable claims. Anything short of that, including a sentence position that happens to look plausible, returns null.

Signal one: a numbered list, bounded 1 through 50

detectListPosition reads genuine numbered-list markup, "1. Fratelli Catering", "2. Bucate pe Roate", and nothing else. It runs on text that has already had rank-prefix markdown noise stripped, so a heading like "### 🥇 1. **Brand**" still resolves correctly. The digit itself is only trusted between 1 and 50: outside that band it is treated as something else entirely, most often a year or a price leading a line, not a rank.

That guard exists because of a real case in our own test data: "2019. Bucate pe Roate a fost premiată în acel an" ("2019. Bucate pe Roate was awarded that year"). Without the band, the leading "2019." reads exactly like a numbered-list marker, and the brand would have scored rank 2019 rather than the correct answer, mentioned but not ranked at all.

Signal two: a bulleted list, but only if the lead-in says so

Bullets are harder, because markdown bullets carry no inherent order the way a numbered list does. detectBulletPosition collects each bulleted block, then checks the few lines immediately above it for one of 25 explicit ordering phrases, "ranked," "in order of," "from best to," "în ordinea," "clasament," before it will read bullet position as rank at all.

17 separate counter-phrases override that check even when an ordering word is also present: "a few," "several," "some of," "no particular order," "câteva," "fără o anumită ordine." An answer that says "here are a few of the top-rated options" contains the word "top," but the "a few" framing wins, and the list scores no rank, correctly, even if the brand happens to be the first bullet.

Why both lists exist

A single "contains a ranking word" check would have been simpler to write and wrong more often. Real AI answers routinely use ranking-adjacent language, "top," "best," "leading," to describe calibre rather than order: "top firms in the area" describes quality, it does not promise the list below it is sequenced. Requiring an explicit ordering phrase, and letting an explicit disclaimer override it, is the narrower and more defensible bar.

Signal three: a stated superlative, anchored to a copula

The third signal covers prose rather than lists: an engine that names a single brand as the top choice in a sentence, without using a list at all. "Cea mai recomandată firmă de catering este Bucate pe Roate" ("The most recommended catering firm is Bucate pe Roate") is a real, checkable claim of rank 1, just not a list-shaped one.

detectSuperlativeRank only accepts this when the superlative is grammatically predicated of the brand through a copula, "este," "is," "remains," sitting immediately before or after the brand name. Co-occurrence in the same sentence is not enough, and membership language is explicitly excluded: "Cele mai bune firme includ Bucate pe Roate" ("The best firms include Bucate pe Roate") contains the same superlative and the same brand name, but "includ" signals membership in a set, not the superlative itself, so it correctly scores no rank.

The signal we deliberately rejected: sentence index

An earlier version of this pipeline fell back to sentence position when neither a list nor a superlative fired: if a brand was named in the third sentence of the answer, it scored rank 3. That felt like a reasonable fallback and was wrong on its own terms, because "third sentence to mention the brand" and "third-ranked recommendation" are not the same unit, and averaging them together into one brand_position column silently blended two different measurements into one number.

The fix was to remove the fallback entirely rather than tighten it. A prose mention with no list and no stated superlative now returns null, full stop, whether the brand is named in the first sentence or the last.

Locked in by the test suite

Five assertions in our regression suite exist specifically to keep this decision from quietly regressing: a genuine numbered-list rank is kept, a prose mention with no list scores null rather than its sentence index, a prose mention deep in a long answer still scores null rather than a high sentence number, a rank beyond the top five is still correctly recovered from a numbered list, and the "2019." case scores null rather than rank 2019. All five, alongside the rest of the suite, must still pass before any change to this pipeline ships.

What a null position tells you that a number can't

A visibility score that never returns a null position for a mentioned brand is not more precise than one that does, it is more willing to guess. The honest state of "the brand was mentioned, but no rank was actually claimed" is real information for a marketer deciding where to focus, and collapsing it into a fabricated number, whether that number comes from sentence order or any other proxy, removes information rather than adding it.

The same three questions we raised in our last piece about extraction apply here to rank: does the methodology test against real production answers, does it disclose the fallback it removed and why, and is there a regression suite standing behind the current behavior. A vendor whose scorer always produces a number, never a null, has usually just moved the guessing somewhere less visible.
